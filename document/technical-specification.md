## 1. System Architecture
The application uses a 3-tier architecture with a unified containerized deployment model:
- **Frontend**: Angular SPA (Single Page Application) styled with Tailwind CSS. Includes JWT-based authentication guards & login interface.
- **Backend (API)**: Spring Boot REST API for processing ticket rules and database interactions.
- **Database**: MySQL relational database holding ticket states.

## 2. API Contract
RESTful API endpoints hosted by Spring Boot:
- `POST /auth/login` - Authenticate user and issue JWT.
- `GET /tickets` - Retrieve all tickets. Returns a list of ticket objects. Requires Bearer Token.
- `POST /tickets` - Create a new ticket. Accepts `name`, `description`, `status`.
- `PUT /tickets/{id}` - Update a specific ticket by its ID.
- `DELETE /tickets/{id}` - Delete a specific ticket by its ID.

*Engineering Review Finding:* The endpoints are standard REST. Ensure that `PUT /tickets/{id}` is fully idempotent, meaning multiple identical requests leave the system in the same state. Ensure `GET /tickets` returns an empty array `[]` (not null) when no tickets exist.

## 3. Database Schema
**Table:** `tickets`
- `id`: BIGINT, Auto-increment, Primary Key
- `user_id`: BIGINT, Not Null (Foreign Key mapping to a users table)
- `name`: VARCHAR(255), Not Null
- `description`: TEXT
- `status`: ENUM('New', 'Assigned', 'Done', 'Escalate'), Default: 'New'

*Engineering Review Finding:* Ensure soft-deletion isn't explicitly required, so a hard delete will be executed on `DELETE`. 

## 4. UI/UX Design Context
**Pages/Views:**
- **Login Page**: JWT email/password authentication screen.
- **Dashboard**: Data table rendering all tickets retrieved from `GET /tickets` for the authenticated team. Includes inline action buttons (`Edit`, `Delete`) per row.
- **Create / Update Modal**: Consolidated slide-out panel overlay to create or modify existing tickets. The Ticket ID is auto-generated by the database and read-only in this form.
- **Search**: Built-in client-side filtering on the Angular list.

*Design Review Findings:* 
1. **Empty State:** The dashboard must have a clear empty state indicating "No tickets found. Create your first ticket." when the database is empty.
2. **Error Handling:** If the Spring Boot backend is down, the frontend should gracefully surface an alert such as "Service unavailable. Please ensure the backend is running." rather than hanging indefinitely.
3. **Primary Key Protection:** Users must not be able to manually enter or override the Ticket ID to prevent database collisions.

## 5. Implementation Strategy & Auto-Decisions
- **Scope check**: Adding authentication increases the complexity slightly, but ensures the system is actually viable for team use. Expected 2-day implementation task.
- **Deployment**: We will rely on a single `docker-compose.yml` file to stitch the local Angular Dev Server, Spring Boot App, and MySQL DB together.
